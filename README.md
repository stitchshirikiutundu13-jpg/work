## この半期で自分がどうなりたいか

「自分のタスクに追われるリーダー」から「チーム全体を見て動けるリーダー」へ変わる。

### ロールモデルと理想像

#### ロールモデル

１人目は新人の頃にお世話になった先輩。
- 周囲への気配りに優れ、すぐに声がけをし、一緒に親身になって考えてくれる。\

2人目は現在の協力会社様。
- 技術に対して貪欲で、新しい技術を積極的に活用し、質問に対して丁寧に受け答えしてくれる。知らないことでもすぐに調べて問題解決してくれる技術力の高さがある。\

#### 理想像

チームメンバーに寄り添い、主体的に声がけ・サポートができるリーダーでありながら、技術に貪欲で「こう書くべき」という判断ができるエンジニア。

### 現状と理想のギャップ

| 理想                                       | 現状                                                            | ギャップ                                         |
| ------------------------------------------ | --------------------------------------------------------------- | ------------------------------------------------ |
| メンバーに寄り添い主体的にサポートできる   | 自分の実装タスクで精一杯で日中の声がけができていない            | リーダー業務に割く時間と仕組みがない             |
| タスクを適切に配分しチームで成果を出せる   | 優秀なパートナー様に依存。もう1名の遅れのリカバリもできていない | タスクの見える化と負荷分散の仕組みがない         |
| 問題を早期に発見し上司・お客様に相談できる | アラートが手遅れになりがち。お客様への進捗遅れの相談が怖い      | アラート基準と報告の型が決まっていない           |
| 技術に貪欲で設計・テストが1からできる      | コードを読むのが精一杯。設計・テスト設計が1からできない         | 技術の基礎体力（設計力・読解力・テスト力）が不足 |
| チームの品質を仕組みで支えられる           | コーディング規約なし、チェックリストなし                        | チーム品質を底上げする仕組みがない               |

### ギャップ解消の方針

多くの課題の根本原因は 「自分の実装タスクに追われ、リーダー業務に手が回っていない」 という構造にある。\
これを解消するために、以下の3つを軸に取り組む。

1. チーム運営の改善 → 負荷分散により自分の時間を確保する
2. 技術基礎力の向上 → 実装スピードが上がり、さらにリーダー業務に時間を使える好循環を作る
3. 品質を支える仕組みづくり → 仕組みで品質を担保し、属人的な対応を減らす

## ■ 大目標1: チーム運営の強化 ～メンバーに寄り添い、チーム全体で成果を出せるリーダーになる～

### 背景・課題

- 自分の実装タスクに追われ、日中の声がけ・進捗確認が朝会・夕会のみになっている
- タスク配分が優秀なパートナー様に偏っており、そのパートナー様が抜けた場合にチームが崩壊するリスクがある
- もう1名のパートナー様は技術力が発展途上で、遅れが出ているがリカバリの仕組みがない
- 問題を一人で抱え込み、「何とかなるだろう → 何とかならなかった → 報告」というパターンに陥っている
- お客様と会話はできているが、進捗遅れなど言いにくいことの相談ができていない

### ◇ 行動目標

行動目標1: 日中の定期声がけを習慣化する
- 朝会・夕会に加えて、午前1回・午後1回（各5分程度）、チャットまたは口頭でメンバーに「困っていることはないか」「進捗はどうか」を確認する
- 確認した内容は簡単にメモに残し、対応が必要な場合は当日中にアクションする

行動目標2: タスクの見える化と負荷分散を実施する\
- 各タスクの担当者・ステータス・期限をチーム全員が見える場所（チャットやチケット管理ツール）で管理する
- メンバーの進捗状況を日次で確認し、遅れが出た場合は当日中にタスクの再分配や粒度の調整（大きなタスクを小さく分割して渡すなど）を行う
- 特に技術力が発展途上のパートナー様には、自力で完了できる粒度のタスクに分割して渡すことを意識する

行動目標3: 上司・お客様への報告・相談を早期に行う
- 上司へのアラートは 「予定の2割遅れた時点」 で上げるルールを自分の中で設定し、実践する
- アラート時は 「現状・影響範囲・自分の考えるリカバリ案」 の3点をセットで報告する
- お客様への進捗報告について、良い報告だけでなく、遅れや懸念点も早い段階で共有することを習慣にする。まずは朝会・夕会の場を活用し、「○○に遅れが出そうなので、相談させてください」と切り出す練習をする

### □ 達成基準

- 達成基準1: 日中の声がけ（午前1回・午後1回）を週4日以上実施し、3ヶ月以上継続できている（日々の声がけメモで確認）
- 達成基準2: タスクの見える化の仕組みを構築し、運用を開始している。半期末時点で、担当者・ステータス・期限が常にチーム全員が確認できる状態を維持できている
- 達成基準3: 遅延やリスクが発生した際に、当日中に上司またはお客様へ報告・相談した実績が半期で3回以上ある（遅延がなければそれ自体が成果としてアピールする）

## ■ 大目標2: 技術基礎力の向上 ～設計・テスト・コードリーディングを自立してできるレベルに近づく～

### 背景・課題

- Web開発歴が浅く、今まで改修案件が中心だったため、1から設計・テスト設計を行った経験がほぼない
- コードを読むのが精一杯で、「この場合はこの書き方が良い」という判断基準を持てていない
- コードレビューをAIに頼っており、自分の目で品質を判断する力が弱い
- 協力会社様のように、技術に貪欲で未知の問題にも自ら調べて対応できるエンジニアになりたい

### ◇ 行動目標

行動目標1: 既存コードベースの読解と学習
- 現在のプロジェクトのコードを隔週1回、1時間かけて読み、「なぜこの設計・書き方にしたのか」を自分なりにドキュメントにまとめる
- 疑問点は協力会社様に 「この部分はなぜこう書いたのか」 と質問し、設計意図や書き方の判断基準を学ぶ（月1回以上）
- まとめたドキュメントはチーム内に共有し、チーム全体のナレッジとして蓄積する

行動目標2: 小規模な機能を1から設計・実装する経験を積む
- 半期中に最低1つ、小規模でもよいので新規機能の設計（基本設計 → 詳細設計 → 実装 → テスト）を自分が主担当で行う
- 設計段階で協力会社様やチームメンバーにレビューを依頼し、フィードバックを取り入れる
- 1から設計した経験を通じて、改修しかできない状態から脱却する足がかりを作る

行動目標3: テスト設計力の強化
- 既存のテストコード（jest）を隔週1回読み、テスト観点のパターン（正常系・異常系・境界値・権限チェック等）を自分なりにドキュメントにまとめる
- 新規実装・改修時にテスト観点一覧を先に作成してからテストコードを書く習慣をつける
- まとめたテスト観点はチーム共有ドキュメントとして蓄積し、チーム全体のテスト品質向上に貢献する

### □ 達成基準

- 達成基準1: コードリーディングのまとめドキュメントを半期で12件以上作成し、チーム内に共有している（隔週1回 × 約24週 = 12件）
- 達成基準2: 新規機能の設計 → 実装 → テストを1件以上、自分が主担当で完遂している（レビュー指摘の対応含む）
- 達成基準3: テスト観点のまとめドキュメントを5パターン以上蓄積し、チーム共有ドキュメントとして公開している

## ■ 大目標3: 品質を支える仕組みづくり ～チーム全体の品質を底上げする仕組みを構築・運用する～

### 背景・課題

- コーディング規約が未整備で、コードの書き方がメンバーごとにバラバラになるリスクがある
- 相互レビューの観点やチェックリストがなく、レビュー品質が属人的になっている
- 開発環境のセットアップ手順書はほぼ整備できているが、開発中に遭遇するエラー対応や効率的な開発手法などのTipsが不足しており、ナレッジが属人化している

### ◇ 行動目標

行動目標1: コーディング規約の策定と自動チェックの導入
- チームのコーディング規約ドキュメントを作成する（命名規則、ディレクトリ構成、コンポーネント設計方針など）
- ESLint / Prettier の設定を規約に合わせて整備し、コード保存時に自動フォーマット・自動チェックが走る状態にする
- 協力会社様にもレビューしてもらい、チーム全員で合意した規約として運用を開始する

行動目標2: 開発ナレッジTipsの拡充
- 開発中に遭遇したエラー対応・トラブルシューティング系のTipsを随時まとめ、チーム共有ドキュメントに追加する
- 開発効率化系（便利なコマンド、VSCodeの設定、デバッグ手法など）のTipsを整理し、チームに共有する
- 新メンバーが来た際にも即戦力化できるよう、「よくある質問・よくあるエラー」 としてまとめ、セットアップ手順書と合わせて活用できる状態にする

行動目標3: レビューチェックリストの作成と運用
- コードレビュー時のチェックリストを作成する（セキュリティ・パフォーマンス・可読性・テスト網羅性・規約準拠など）
- レビューで見つかった指摘事項をナレッジとして蓄積し、チェックリストを継続的に更新する
- チェックリストをチーム全員が活用し、「誰がレビューしても一定の品質を担保できる」 状態を目指す

### □ 達成基準

- 達成基準1: コーディング規約ドキュメントを作成・チーム合意を取得し、ESLint / Prettier で自動チェックが動いている状態になっている
- 達成基準2: 開発ナレッジTipsを10件以上蓄積し、チーム共有ドキュメントとして公開・運用されている（エラー対応系・開発効率化系それぞれ含む）
- 達成基準3: レビューチェックリストを作成・運用を開始し、レビュー指摘のナレッジが5件以上蓄積されている

## 効果測定

上記の目標に取り組むことで、以下の効果が期待できる。

| 大目標             | 期待される効果                                                                   | 測定方法                                                         |
| ------------------ | -------------------------------------------------------------------------------- | ---------------------------------------------------------------- |
| チーム運営の強化   | メンバーの困りごとの早期発見・対応、タスク遅延の減少、お客様との信頼関係向上     | 声がけメモの記録数、遅延発生時の報告タイミング（当日中かどうか） |
| 技術基礎力の向上   | 実装スピードの向上、設計・テストの自立度向上、リーダー業務に充てられる時間の増加 | まとめドキュメントの件数、新規設計の完遂件数                     |
| 品質の仕組みづくり | コード品質のばらつき減少、レビュー品質の均一化、属人性の排除                     | 規約の運用状況、チェックリストの活用状況と蓄積件数               |
