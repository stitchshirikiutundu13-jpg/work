# 2025年度下期 目標

## この半期で自分がどうなりたいか

人員の入れ替わりがあってもチームとして同水準の成果を出せる状態を作り、
計画と実績の乖離を抑える工数管理ができるリーダーになる。

## ロールモデル

- 新人時代の先輩: 周囲への気配りに優れ、メンバーの自立を促す育成姿勢
- 現在のパートナー様: 新しい技術への貪欲さと、自ら調べて解決する技術力

## 現状の課題

- 優秀なパートナー様が6月末で離脱、7月に新規メンバーが参加予定
- 1月末参加のパートナー様は主要Webアプリの仕様理解が浅く、初心者レベル
- 仕様・観点・規約が属人化しており、メンバーが入れ替わると品質が落ちる構造
- 進捗管理が属人的で、計画と実績の乖離を定量で把握できていない
- 前期は2案件で遅延ゼロだが、私の残業が月平均35h・ピーク80hと持続不可

## 半期のフェーズ

- 4月〜6月: 集中引き継ぎ期間（仕様説明、ペアプロ、Tips整備、2次レビュー体制）
- 7月〜9月: 自立運用期間（チェックリスト運用、私が1人でレビュー、新規メンバー受け入れ）

## 大目標1: 人員が入れ替わってもチームとして同水準の成果を出せる状態を作る

### 行動目標と達成基準

行動目標1: 6月末までに、主要機能の仕様説明セッションとペアプロ・コードリーディングを集中実施し、パートナー様への知見引き継ぎを行う
- 達成基準1: パートナー様が中規模タスク（工数3人日以上）を設計→実装→テストまで一人で完遂した件数が1件以上ある

行動目標2: 開発ナレッジTips（エラー対応、効率化、Web開発で学んだ技術トピック、環境固有の注意点）を共有ドキュメントに整備する
- 達成基準2: 共有ドキュメントに登録したTipsが10件以上ある（うち技術トピック系3件以上）。かつ半期内にメンバーが参照・利用した記録（コメント／チャット引用／レビュー指摘での参照）が3件以上残っている

行動目標3: 設計観点・テスト観点・コーディング規約を整理してレビューチェックリスト化し、6月末までは1次／2次レビューの指摘差を記録して観点漏れを潰す
- 達成基準3: 半期で対応した全案件のバグ密度が一般指標の範囲内に収まる（結合試験: 0.5〜1.0件/KStep、総合試験: 0.1〜0.3件/KStep）

行動目標4: 7月参加の新規メンバー受け入れ手順（環境構築、仕様キャッチアップ、初期タスクの渡し方）を手順書として整備する
- 達成基準4: 7月参加日までに手順書を完成させ、新規メンバーが参加から10営業日以内に初期タスク（5人日以内のもの）へ単独で着手できている

## 大目標2: 計画と実績の乖離を低減し、工数管理を仕組み化する

### 行動目標と達成基準

行動目標1: 案件着手時に見積もりフォーマット（全体工数／タスク分解／アサイン／バッファ／依存関係／リスク）を作成し上司レビューを受ける
- 達成基準1: 半期で対応した全案件で見積もりフォーマットを作成し、上司レビュー済みである

行動目標2: 週次で予実管理を行い、計画と実績の差分・対応策を記録する。予定の2割遅れを検知した時点で上司・お客様にアラートを上げる
- 達成基準2: 全週分の予実記録と、遅延発生時のアラート・対応策の記録が残っている。半期で対応した全案件において、案件完了時点の計画工数と実績工数の乖離率が±15%以内に収まっている。遅延件数ゼロを維持する

行動目標3: 案件終了ごとに振り返りを行い、乖離要因と再発防止策を見積もりフォーマットに反映する
- 達成基準3: 半期で2案件以上の振り返り記録があり、見積もりフォーマットの改訂履歴が1回以上残っている

行動目標4: 残業時間・メンバー稼働を月次で記録し、改善状況を数値で追う
- 達成基準4: 会社業務を除く私の残業を月平均30h以下／ピーク40h以下に抑える（前期: 月平均35h／ピーク80h）。メンバー残業を月平均25h以下に維持する
